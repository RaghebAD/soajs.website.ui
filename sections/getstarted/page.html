What is SOAJS?

SOAJS refers to Service-Oriented Architecture built using JavaScript.
Service-Oriented architecture (SOA) is a design pattern in which application components provide services to other components via a communications protocol, typically over a network. The principles of service-orientation are independent of any vendor, product or technology.

Services can be combined to provide the functionality of a large software application. SOA makes it easier for software components on computers connected over a network to cooperate. Every computer can run any number of services, and each service is built in a way that ensures that the service can exchange information with any other service in the network without human interaction and without the need to make changes to the underlying program itself.

This makes SOAJS self-aware of all services running in its cloud and can be used to create APIs between the Frontend and Backend of a product without the need of modifying the code of that product.

Infrastructure

Before we start, let's take a look at the infrastructure of SOAJS and understand what we will be dealing with.

SOAJS uses a gateway architecture meaning all requests go through one entry point before being processed. This entry point is referred to as the Controller and all requests pass through that Controller and then get forwarded to their corresponding service after undergoing some security checks. Some of the requested services are protected by oAuth, in that case, when the request reaches the Controller, it is first forwarded to the oAuth service for validation and if valid, it is then forwarded to the corresponding service.
Now that you know how SOAJS functions, go ahead install it and checkout the examples to learn how to build APIs super fast.

Install SOAJS

To help you get started, we have created 3 examples that show you the power and easiness of SOAJS and how to build APIs super fast. First thing you need is to install SOAJS on your machine. You can do that using 2 methods:

Docker Installation

If you are familiar with Docker, you can use it to install SOAJS on your machine.
Docker installs SOAJS on your machine, sets up the needed environment & dependencies and also loads a pre-configured sample data in the database so you can start working immediately. 
			$ docker install soajs(this line will be updated once docker is completed)

Manual Installation

  1. Prerequisites

    Before you install SOAJS, make sure you have the following technologies installed:
    - MongoDB ( version 2.6.7 or higher)
    - NodeJS (version 0.10.25 or higher)

  2. Installing SOAJS

    SOAJS in an NPM repository and can be installed using NPM command in your terminal.
    If you have NodeJS installed, then you can open your terminal and execute the following commands: 
    	                #install soajs services
$ npm install soajs.util

  3. Start the services

    To start a service, using the command line, navigate to its folder, and run node index.
    	                # navigate to soajs.util/controller directory
$ cd soajs.util/controller;
# start the service
$ node index
    Checkout the Controller section under documention for more information.
    Repeat this step in all other services: oAuth, URAC and examples 01 - 02 - 03 -04 to get them running.

  4. Testing the Service

    By default, the controller runs on your localhost via port 4000. Every service in SOAJS has its own default port and another maintenance port configured in the registry. So when you start a service, the service will be listening to two ports at the same time; one to process the requests and another used for maintenance operations.
    You can access the maintenance port while the service is running and perform some maintenance operations like heartbeat checking and reload configuration. To access the maintenance port of a service, add 1000 to its default port number, in this case 4000 + 1000 = 5000.
    Let's see what happens when we run both 4000 and 5000 ports on your machine.

    Default Port 
    $ curl -X GET "http://localhost:4000
    Response:
    {
	"result": false,
	"errors": {
		"codes": [ 130 ],
		"details": [
			{
				"code": 130,
				"message": "Unknown service."
			}
		]
	}
}
    When you invoke the controller on the default port and without specifying a service, the controller simply returns an error message.


    On its own, the controller is useless but in this case, we just wanted to check that there is a response and verify that it is running and listening on port 4000.

    Maintenance Port

    Let's hit the maintenance port and perform a heartbeat check.

    $ curl -X GET "http://localhost:5000/heartbeat
    Response:
    {
	"result": true,
	"ts": 1425041775082,
	"service": {
		"service": "CONTROLLER",
		"type": "rest",
		"route": "/heartbeat"
	}
}
    You can perform a heartbeat check on any service using this approach. Simply call the maintenance port of a service and invoke the "/heartbeat" api to check if it is running.


    Great! the service is healthy and running as expected.

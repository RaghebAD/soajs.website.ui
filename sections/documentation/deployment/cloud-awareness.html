Cloud Awareness

In Overview we showed the below diagram and mentioned the technologies needed to get SOAJS up and running on either one machine or a cloud. In this Section we will demonstrate how to install SOAJS and these needed technologies as well as how to configure them on different nodes creating a cloud.
SOAJS is designed to work in a cloud environment meaning its services should be spread on different nodes, therefore we need multiple servers at our disposal and this can accomplish this via virtualization. In this section we will not discuss which virtualization should be used to create the servers, this is left up to you to pick the technology you desire, we will focus more on how many servers we need and for what exactly.


Cloud Architecture

The cloud architecture is split up to two Environments: Dashboard - Custom.
The Dashboard Environment is used to manage the content of core databases such as provisioned tenants, products and environments. The Dashboard also manages the users that have access to this environment ( members of the administrator group ). This environment is independent from the custom environments and the database layer. This environment contains a list of nodes that provided access and services to the database to managed the mentioned data types.


The Custom Environment on the other hand summarizes three environments that you might use while developing any project:

  • Staging: This environment is used by quality engineers to test the updates and fixes that are pushed by developers after completion. Quality Engineers deploy the update on an environment that resembles the live production environment and conduct several testing approaches and cases to make sure the update works correctly in all scenarios before signing it off.
  • CAT: This environment is known as staging or client approval test and is basically a replica of production. Once an update is pushed by a quality engineer, it is deployed on this environment for the client to see and approve it.
  • Production: This is live environment that the client uses to interact and serve his customers. Updates that he approves from CAT are deployed on this environment and become live for the public.

There are cases where not all three environments are used depending on the project type and size but what is important to understand is the similarity between these three environments. In terms of cloud architecture, they are basically the same but their usage is different; production environment might have more nodes than the other two since it servers public users and will scale based on traffic needs whereas the other two are restricted between the development company and the client himself.
Because these environments are similar, we will pick one of them (Production) and show how to get such an environment up and running with all of SOAJS services distributed.


Installation:

Our aim is to install the database servers, Dashboard and Production environment on separate servers and create the cloud of SOAJS services. This requires several steps and configuration that are grouped under 3 categories: Database - Dashboard Environment - Production Environment.

We begin first by installing the databases and configuring them to work as replica sets. We will require two sets of databases each replicated over 5 servers and serving a specific purpose. The first set is referred to as CORE and is used to manage the provisioned information of tenants, products, environments, dashboard members and their sessions. The second set is referred to as Prod and contains tenant specific information as well as any other additional custom services databases information.

The image above illustrates how we will construct the database system of our cloud. You can always add additional Replica Sets to distribute the load and expand. Your custom services databases do not need to be in the same set with the tenants databases, you can split those as well. Expand your database based on your need, for the sake of this tutorial we chose to group Core databases in one group and other databases in another group resulting in two sets.

Start by creating 2 Virtual Machines (VMs) and install mongodb on each of them. Give your Virtual Machines fixed IP addresses, we will connect to these two VMs later on, for this tutorial we will use two local IP address: 192.168.0.101 - 192.168.0.102 and we will refer to them as mongo-core and mongo-prod. 

#install MongoDB latest stable version on mongo-core VM
$ apt-get install mongodb

# edit /etc/hosts of mongo-core VM
127.0.0.1 mongo-core

#install MongoDB latest stable version on mongo-prod VM
$ apt-get install mongodb

# edit /etc/hosts of mongo-prod VM
127.0.0.1 mongo-prod
Once mongo is installed, create a replica set of 5 databases on each of these machines. If you are not familiar with Replica Sets, visit the official mongodb replica set documentation. Our first replica set will run on the first VM and will contain the CORE databases, the second replica set will run on the second VM and will contain the production database. Complete the steps below on each of the VMs, to get both replica set up and running:

By now both VMs have mongo installed and running as replica set, navigate to mongo-core VM, install soajs.mongodb.data and import the core sample data to the database. 

# install soajs.mongodb.data
$ npm install soajs.mongodb.data

# import the core sample data
$ cd soajs.mongodb.data/modules/dashboard-ui/
$ chmod +x import.sh
$ ./import.sh
The shell script will create 1 tenant, 1 product, 1 user and 1 group in our core database and will replicate the information on all members of the replica set we created. We will use this sample data later on when everything is running. After executing the script connect to mongo and list the databases to verify the last step. 
# connect to mongo on mongo-core VM
$ mongo

# show the databases
mongo > show dbs
core_provision 0.078GB
local 0.078GB
dashboard_urac 0.078GB
admin (empty)

# switch to core_provision db
mongo > use core_provision

# list collections in the db
mongo > show collections
environment
oauth_token
oauth_urac
products
system.indexes
tenants
The above command will list the databases and the collections of core_provision inside the mongo-core VM. This database information is used by SOAJS services and is considered as the core data.

The dashboard environment consists of a set of servers that communicate with one another to provide access to the mongo-core VM which contains the core database. The set of servers is also replicated for load balancing and traffic distribution. Having that said, we will create a VM and install Nginx on it. We will instruct Nginx to upstream 3 servers that contain the controller of SOAJS which is the main gateway so that all requests arriving to the dashboard get handled by 3 servers.
The Environment will contain an additional server that runs the dashboard_urac, oAuth and dashboard services. The dashboard_urac service is basically a Urac service but specific for the dashboard environment only.
The image below demonstrate what the Dashboard Environment will look like when we complete this step.

Every server in this environment contains HAproxy that points to the other servers. HAproxy along with SOAJS heartbeat provide awareness among the cloud between the services.
When a request arrives to the controller, the controller forwards this request to the requested service but since none of the services is installed on the same server, HAproxy tells the controller where to forward this request to the correct server that contains this service as marked by the red arrows. These servers are replicated therefore the same thing happens on all 3 controllers.

In another event, when a service in this cloud wants to communicates to another service, it does that via the controller, again HAproxy that resides on the same server of the calling services, tells the service which server the controller is located on and the request is sent to that server as marked by the blue arrows, then the controller redirects the request using the same technique mentioned in the previous paragraph back to the requested service as marked by the red arrows. These servers are also replicated therefore the same thing happens on all 3.

Servers Setup:

Now that we know what will be deploying, let's go ahead and create these servers. Start by creating a VM and install Nginx on it, then use the configuration located in soajs.utilities/deployment/cloud/dashboard/ to tell Nginx how many controllers we will have and where they are. For this demo, we will use the following list of IP addresses:

  • Nginx: 192.168.0.50
  • Controller 1: 192.168.0.51
  • Controller 2: 192.168.0.52
  • Controller 3: 192.168.0.53
  • Services Server 1: 192.168.0.61
  • Services Server 2: 192.168.0.62
  • Services Server 3: 192.168.0.63

Nginx Setup:

Once you have created all 7 VMs, install Nginx using below the command on the Nginx VM 192.168.0.50: 

# install Nginx latest stable version
$ apt-get install nginx
When Nginx is installed, locate its configuration file; On Ubuntu the configuration file is located in /etc/nginx/nginx.conf. Open that file in edit mode, scroll to the bottom and instruct nginx to also look at the soajs.utilities/deployment/cloud/dashboard/nginx/conf/ folder and load all the configuration files in it, add the line before the closing bracelet of http inside the nginx.conf. 
http {
	# default nginx configuration goes here
	include /opt/soajs/soajs.utilities/deployment/cloud/dashboard/nginx/conf/*.conf;
}
This line will now load the Nginx configuration that SOAJS needs when Nginx starts. Inside the conf folder, you will find a file named upstream.conf. If you have chosen different IP addresses for your controllers, make sure you updated the entries in that file.
Edit your hosts file and add 2 entries to point out that these domains are on your local machine. 
$ sudo gedit /etc/hosts
When the hosts file is loaded in gEdit editor, add the following: 
127.0.0.1 api.soajs.org
127.0.0.1 dashboard.soajs.org
You can also copy these entries from soajs.utilities/deployment/cloud/dashboard/hosts to your /etc/hosts file.
Go ahead and Start Nginx Now: 
$ sudo Nginx
When Nginx starts, test the configuration you just created by opening your browser and invoking 2 domains: api.soajs.org and dashboard.soajs.org. Nginx will reply with a 502 Bad Gateway Error instead of Server not Found. Till this point, this is normal because we haven't started any service yet, but the important thing is that Nginx is now redirecting the requests to a local SOAJS server assuming that the later is up and running.


Services Setup:

Now let's install the services on the remaining 6 VMs; 3 of them will contain controllers and 3 will contain oAuth, Urac and Dashboard Services.
Note: you can always create 2 VMs, install the services on them, then clone those VMs and change their IP addresses.

On each controller VM, install nodejs and HAproxy, then install the controller and the utilities of SOAJS.
Execute the below commands in your terminal: 

# install haproxy
$ apt-get install haproxy

# install nodejs
$ apt-get install nodejs

# create folder path to put all soajs components in it
$ mkdir -p /opt/soajs/node_modules/

# install SOAJS controller and utilities
$ npm install soajs.controller
$ cd soajs.controller

$ npm install soajs.utilities
The next step requires to configure HAproxy to tell the controllers what are the IP address of the services that we have in the cloud.
First create a backup of the default HAproxy file in /etc/haproxy/haproxy.cfg, then Navigate to soajs.utilities/deployment/cloud/dashboard/ and copy the file dashboard-controller-haproxy.cfg to /etc/haproxy/ and rename it to haproxy.cfg.
Keep in mind that if you have chosen a different IP set for the services, you need to edit this file and change the IP address in it to meet your configuration.

Once the controller VMs are completed, we need to set up the remaining services. In the diagram above we have 3 servers and we gave them IP addresses 192.168.0.61/62/63. On each service VM, install nodejs and HAproxy then install the utilities, urac, oauth and dashboard services of SOAJS.

# install haproxy
$ apt-get install haproxy

# install nodejs
$ apt-get install nodejs

# create folder path to put all soajs components in it
$ mkdir -p /opt/soajs/node_modules/

# install soajs.utilities
$ npm install soajs.utilities

# install soajs.urac
$ npm install soajs.urac

# install soajs.oauth
$ npm install soajs.oauth

# install soajs.dashboard
$ npm install soajs.dashboard

You also need to configure HAproxy on these 3 VMs to point to the controller VMs. Again locate HAproxy configuration file and create a backup copy of /etc/haproxy/haproxy.cfg, then Navigate to soajs.utilities/deployment/cloud/dashboard/ and copy the file dashboard-services-haproxy.cfg to /etc/haproxy/ and rename it to haproxy.cfg.
If you chose to use different IP addresses than the ones in this tutorial, keep in mind you need to update this configuration file as well and set the correct IP addresses values.

Now point out the database server so that both the controllers and the services can connect to mongo-core. So in all 6 VMs we created last, edit the /etc/hosts file and add the following entry: 

# edit /etc/hosts
192.168.0.101 mongo-core

By now you have completed 2/3rd of setting up this environment and you should have something that resembles the following diagram:

Notice the purple arrows as they point out that all our servers in the dashboard environment connect to the database server.
Here is a summary of list of VMs we used for this environment and their IP addresses:

  • mongo-core: 192.168.0.101
  • Nginx: 192.168.0.50
  • Controller 1: 192.168.0.51
  • Controller 2: 192.168.0.52
  • Controller 3: 192.168.0.53
  • Services Server 1: 192.168.0.61
  • Services Server 2: 192.168.0.62
  • Services Server 3: 192.168.0.63

The final step to complete this environment setup is to use the correct SOAJS environment project. As mentioned in the Registry, SOAJS supports multiple projects, and in this case we need to use the dashboard project that contains the environment variables that we need for this cloud using the above IP addresses from the tutorial. The dashboard project is located in soajs.utilities/deployment/cloud/dashboard/profiles/dashboard.
In all 6 VMS where we installed soajs.utilities, create 3 environment variables and give them the following values: 

# set the environment name to dashboard
$ export SOAJS_ENV=dashboard

# set the project name to dashboard
$ export SOAJS_PRJ=dashboard

# point out the location of the project
$ export SOAJS_REGDIR=/opt/soajs/node_modules/soajs.utilities/deployment/cloud/dashboard/

Once you completed all above steps, start all the services in all 6 VMs and the Dashboard Environment will run.

The Production environment consists of a set of servers that communicate with one another to provide access to the mongo-prod VM which contains the data databases that are used by tenant services and your custom made services. The set of servers is also replicated for load balancing and traffic distribution. The setup is close to the above dashboard environment with one difference, we don't need the dashboard service here.
We will create a VM and install Nginx on it. We will instruct Nginx to upstream 3 servers that contain the controller of SOAJS which is the main gateway so that all requests arriving to the services get handled by all 3 servers.
The Environment will contain an additional server that runs the urac, oAuth and your custom services. The urac service in this case is tenant driven this means that every tenant you create will have its own database, its own users and groups therefore, in this tutorial we will refer to it as tenant_urac when it comes to this environment.
The image below demonstrate what the Production Environment will look like when we complete this step.

As you can see, the setup is quite close to what we did in the dashboard environment. We made one difference here, we chose to use 9 servers: 3 for controllers, 3 for soajs services and 3 for your custom services. You can merge the servers that contains soajs services with your custom services if you want to reduce the numbers of servers from 9 to 6.

Notice that here as well, the servers are replicated and that HAproxy is present and used to tell point out the servers IP addresses for the controllers and our services as demonstrated by the blue and red arrows. Here is the list of IP addresses we will use for this environment:

  • mongo-prod: 192.168.0.102
  • Nginx: 192.168.0.70
  • Controller 1: 192.168.0.71
  • Controller 2: 192.168.0.72
  • Controller 3: 192.168.0.73
  • SOAJS Services Server 1: 192.168.0.81
  • SOAJS Services Server 2: 192.168.0.82
  • SOAJS Services Server 3: 192.168.0.83
  • Custom Service Server 1: 192.168.0.91
  • Custom Service Server 2: 192.168.0.92
  • Custom Service Server 3: 192.168.0.93

Go ahead and create all these VMs with the exception of the mongo-prod, we already did that. Install Nginx on the first VM and for the remaining 9 VMS install haproxy and nodejs on all of them. You will also need to create the soajs directory and install soajs.utilities on them all.

Once you have created all 10 VMs, install Nginx using below the command on the Nginx VM 192.168.0.70:

Open that file in edit mode, scroll to the bottom and instruct nginx to also look at the soajs.utilities/deployment/cloud/prod/nginx/conf/ folder and load all the configuration files in it, add the line before the closing bracelet of http inside the nginx.conf.

	include /opt/soajs/soajs.utilities/deployment/cloud/prod/nginx/conf/*.conf;

This line will now load the Nginx configuration that SOAJS needs when Nginx starts. Inside the conf folder, you will find a file named upstream.conf. If you have chosen different IP addresses for your controllers, make sure you updated the entries in that file.
Edit your hosts file and add the following entry to point out the domain of the the controller. 
$ sudo gedit /etc/hosts

You can also copy these entries from soajs.utilities/deployment/cloud/prod/hosts to your /etc/hosts file.
Go ahead and Start Nginx Now: 
$ sudo Nginx
When Nginx starts, test the configuration you just created by opening your browser and invoking the domain: api.soajs.org. Nginx will reply with a 502 Bad Gateway Error instead of Server not Found. Till this point, this is normal because we haven't started any service yet, but the important thing is that Nginx is now redirecting the requests to a local SOAJS server assuming that the later is up and running.


Services Setup:

Now let's install the services on the remaining 9 VMs; 3 of them will contain controllers and 3 will contain oAuth and Urac and 3 will contain your services.
On each controller VM, install nodejs and HAproxy, then install the controller and the utilities of SOAJS.
Execute the below commands in your terminal: 

# install haproxy
$ apt-get install haproxy

# install nodejs
$ apt-get install nodejs

# create folder path to put all soajs components in it
$ mkdir -p /opt/soajs/node_modules/

# install SOAJS controller and utilities
$ npm install soajs.controller
$ cd soajs.controller

$ npm install soajs.utilities
The next step requires to configure HAproxy to tell the controllers what are the IP address of the services that we have in the cloud.
First create a backup of the default HAproxy file in /etc/haproxy/haproxy.cfg, then Navigate to soajs.utilities/deployment/cloud/prod/ and copy the file prod-controller-haproxy.cfg to /etc/haproxy/ and rename it to haproxy.cfg.
Keep in mind that if you have chosen a different IP set for the services, you need to edit this file and change the IP address in it to meet your configuration.

Once the controller VMs are completed, we need to set up the remaining services. In the diagram above we have 6 servers and we gave them IP addresses 192.168.0.81/82/83/91/92/93. On each service VM, install nodejs and HAproxy then install the utilities, urac, oauth and your service.

# install haproxy
$ apt-get install haproxy

# install nodejs
$ apt-get install nodejs

# create folder path to put all soajs components in it
$ mkdir -p /opt/soajs/node_modules/

# install soajs.utilities
$ npm install soajs.utilities

# install soajs.urac
$ npm install soajs.urac

# install soajs.oauth
$ npm install soajs.oauth

# install your_service
$ npm install your_service

You also need to configure HAproxy on these 6 VMs to point to the controller VMs. Again locate HAproxy configuration file and create a backup copy of /etc/haproxy/haproxy.cfg, then Navigate to soajs.utilities/deployment/cloud/prod/ and copy the file prod-services-haproxy.cfg to /etc/haproxy/ and rename it to haproxy.cfg.
If you chose to use different IP addresses than the ones in this tutorial, keep in mind you need to update this configuration file as well and set the correct IP addresses values.

Now point out the database servers so that both the controllers and the services can connect to mongo-core and to mongo-prod. In this environment, the services need to read from core and from the prod databases. In the dashboard environment we only needed to connect to core. The logic behind this strategy is that when a request is made and a tenant key is provided for example, the tenant records are located in core and to cross reference the key value, data from core need to be accessible.
So in all 9 VMs we created last, edit the /etc/hosts file and add the following entry: 

# edit /etc/hosts
192.168.0.101 mongo-core
192.168.0.102 mongo-prod


By now you have completed 2/3rd of setting up this environment and you should have something that resembles the above diagram.

Notice the purple arrows as they point out that all our servers in the production environment connect to the production database server and the green arrows point out the connection to the core database server.

The final step to complete this environment setup is to use the correct SOAJS environment project. As mentioned in the Registry, SOAJS supports multiple projects, and in this case we need to use the prod project that contains the environment variables that we need for this cloud using the above IP addresses from the tutorial. The prod project is located in soajs.utilities/deployment/cloud/prod/profiles/prod.

	"extKeyRequired": true,
	"port": 4003,
	"host": "127.0.0.1",
	"url": "http://127.0.0.1:4000/your_service"
}
Note: the varialbe your_service is also present at /opt/soajs/node_modules/soajs.utilities/deployment/cloud/prod/prod-controller-haproxy.cfg, make sure you update the value as well.
Once you completed all above steps, start all the services in all 9 VMs and the Production Environment will run.
